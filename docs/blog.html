What is Khalil

Khalil is a personal AI agent that runs as a macOS daemon. It connects to email, calendar, terminal, IDE, and 200+ other actions through a Telegram interface. It indexes 46,000 documents into a local vector database, uses Claude/GPT/Ollama for reasoning, and can generate artifacts (files, presentations, scripts) from its knowledge base.

When Khalil detects a capability gap — a request it can't handle — it generates a new action module, tests it in a sandbox, and opens a PR against its own codebase. 30+ capabilities were built this way.

The initial architecture

The first design was a flat tool-routing loop: message arrives, LLM picks a tool, tool executes, response returns.

Message → LLM → Pick tool → Execute → Response

This worked for single-turn requests — weather, reminders, email search. It broke on anything multi-step.

The failure that exposed the architecture gap

The request: "Build me an HTML presentation about Fall 2026 planning." The knowledge base had all the relevant documents (46,000 indexed from email, Drive, repos, notes).

What happened: the LLM called search_knowledge 10 consecutive times across 36 hours. Zero files created. It never called generate_file.

The root cause is a known LLM behavioral pattern: research is low-risk (always produces results), generation is high-risk (might produce a bad file). Given unlimited iterations with the same tools available, the model defaults to the safe action indefinitely.

This is not a prompting problem. It's an architectural problem — there was no mechanism to enforce a transition from research to action.

The redesign: phase-aware execution

The fix required restructuring the pipeline into explicit phases with different rules per phase.

Intent classification

Before any LLM call, a heuristic classifier (regex + pattern matching, no LLM needed) categorizes the message:

  • TASK: "Build me a presentation" → full pipeline
  • QUESTION: "What's the weather?" → search + answer
  • CONTINUATION: "Yes", "Sounds good" → inherit active task
  • CHAT: "Hello", "Thanks" → conversational only

This fixed a class of bugs where "Yes" (to continue a task) triggered a KB search on the literal word "Yes", returning irrelevant documents.

Intent-aware context assembly

Each intent type assembles different context. CONTINUATION skips KB search and uses task state + 5 recent messages. TASK gets deep retrieval (KB search + full document read). CHAT gets conversation history only.

Research cap and escalation

A PhaseTracker monitors tool usage per iteration and enforces an escalation ladder for artifact tasks:

Free Researchiter 0-3
All tools available. Search, read documents, explore. No restrictions.
Nudgeiter 4
Prompt injected: "You have enough context. Call generate_file NOW."
Restrictiter 5
Search tools physically removed from the tool list. Only action tools remain.
Forceiter 6+
tool_choice locked to generate_file. The LLM cannot call any other tool.
Programmatic Fallbackexhaustion
LLM bypassed entirely. System constructs the generate_file call from gathered tool results.

Each level removes more LLM autonomy. At level 3, the only available tool is generate_file. At the programmatic fallback, the system constructs the tool call directly from gathered results — the LLM is not consulted.

Verification

Post-action verification checks that side effects actually occurred. If generate_file reports success, the system confirms the file exists on disk and has content. Task state persists across messages in a DB table, with auto-reset after 3 consecutive failures.

Failure domain isolation

With the phase-aware architecture in place, the system still failed — this time due to infrastructure contention.

The observed failure chain: every save_message() call triggered a background summarization job that made its own LLM API call. Between tool-use iterations, 2-3 background tasks would fire through the same HTTP client. When they timed out, they tripped a shared circuit breaker — which then blocked the foreground user request.

The user's request failed because a background housekeeping task consumed the API's error budget.

The fix: failure domain isolation.

  • Separate circuit breakers for foreground (user requests, threshold=5) and background (summarization, threshold=2)
  • Suppress background LLM calls entirely during active tool-use loops
  • Separate connection pool for long-running file generation
  • Automatic retry for transient errors with escalating timeouts

Result

After these changes, the same request — "Build me an HTML presentation about Fall 2026 planning" — produced a 704-line file in 2 minutes on the first attempt. generate_file was called on iteration 0. The phase tracker, context assembly, and verification layer all functioned as designed.

The system went from 5 failed attempts over 2 days to reliable execution. The difference is structural: the architecture prevents the failure modes rather than patching individual symptoms.

Quality regression prevention

To prevent degradation from this baseline, the system uses a 4-tier quality framework:

  • CI gates: Every PR runs 77 tests including 24 behavioral contracts — formalized invariants like "the research cap must be 4 iterations" and "background circuit breaker failures must not trip the foreground breaker."
  • Post-restart smoke test: Every time Khalil restarts, it verifies the pipeline is wired correctly. Intent classifier works? PhaseTracker loads? Core tools present? All checked in under 1 second.
  • Metric baselines: Frozen thresholds for tool success rate (>80%), latency P95 (<60s), and error cascade rate (<10%).
  • Behavioral contracts: 24 tests that encode invariants, not implementation details. You can refactor the PhaseTracker however you want — the contract still holds.

Observations

01
Architecture > patches

I spent 3 days adding patches to a flat tool-routing loop. Each patch introduced new edge cases. The architectural redesign (intent classification + phase-aware execution) solved 7 failure modes at once.

02
Background tasks are invisible killers

The hardest bug to find was background summarization tripping a circuit breaker that killed user requests. The fix (failure domain isolation) is infrastructure work that's unglamorous but essential.

03
LLMs need structure, not freedom

Given unlimited tool iterations, the LLM will research forever. The escalation ladder (nudge → restrict → force → programmatic) is the key insight: progressively remove the LLM's ability to procrastinate.

04
Behavioral contracts > unit tests

Unit tests verify implementation. Contracts verify invariants. When you refactor, unit tests break (by design). Contracts survive — they test what must be true, not how it's implemented.

05
Self-improvement is real, but fragile

Khalil generated 30+ capabilities autonomously. But the evolution engine needs guardrails — CI gates, eval suites, sandbox execution. Without them, self-improvement becomes self-destruction.

06
The demo is the easy part

Getting "Build me a presentation" to work once is a weekend project. Getting it to work reliably — across API outages, timeout cascades, LLM behavioral quirks — took 2 weeks of architectural work.
